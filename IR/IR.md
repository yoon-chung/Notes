# IR

## 1. 검색엔진 발전과정

### 1.1 문제 정의

Information Retrieval(IR)이란 대규모 비정형 텍스트 문서 집합에서 사용자의 질의(query)에 적합한 문서를 찾아 정렬하여 반환하는 과정이다.

- **입력**: 사용자의 키워드 또는 텍스트 질의
- **대상**: 비정형 문서(주로 텍스트)로 구성된 대규모 문서 집합
- **출력**: 적합도 순으로 정렬된 문서 목록

### 1.2 핵심 이슈: 속도와 품질

**속도 측면의 과제**
- 문서 규모가 극단적으로 커질 때의 대응
- 실시간 문서 추가/삭제/업데이트 처리
- 복합 질의, 필터링 조건, 높은 트래픽 상황에서도 1초 이내 응답

**품질 측면의 과제**
- 질의-문서 간 적합도를 어떻게 수치화할 것인가
- 문서 자체의 품질을 어떻게 반영할 것인가
- 사용자 피드백을 랭킹에 어떻게 활용할 것인가

### 1.3 데이터 구조: 역색인 (Inverted Index)

문서 → 단어가 아니라 **단어 → 문서 목록** 형태로 색인을 구축하는 방식이다. 특정 단어가 포함된 문서를 빠르게 조회할 수 있어 검색 속도를 획기적으로 개선한다.

| Term | Documents |
|------|-----------|
| 검색 | doc1, doc3 |
| 랭킹 | doc2 |


### 1.4 랭킹 알고리즘

검색된 문서를 적합도 순으로 정렬하기 위한 점수 산출 체계로, 크게 세 가지 요소로 구성된다.

1. **질의-문서 유사도**: TF-IDF, BM25 등을 활용하여 질의와 문서 간 연관도 점수 계산
2. **문서 자체 품질**: PageRank, spam score, recency 등 질의와 독립적인 품질 지표
3. **Relevance Feedback**: 클릭률, 체류 시간 등 암묵적(implicit) 사용자 피드백 활용

### 1.5 새로운 도전 과제

역색인과 랭킹으로 큰 발전을 이뤘지만 여전히 남은 문제들:

- **성능 최적화**: 문서, 사용자 관심사, 언어 자체가 계속 변화하므로 형태소 분석기와 랭킹 모델의 지속적 유지보수 비용 발생
- **컨텍스트 활용의 한계**: 키워드 입력 → 결과 확인의 반복적 인터페이스를 넘어 자연어 질의와 개인화가 필요

### 1.6 인공신경망의 적용

**Vector Similarity (임베딩 기반 유사도)**

Transformer 등 신경망 모델로 텍스트를 벡터로 변환(embedding)한 뒤, 벡터 간 유사도를 계산하여 의미적으로 유사한 문서를 찾는 방식이다. 키워드가 다르더라도 의미가 같으면 매칭이 가능하다.

- 짧은 쿼리(의도 모호) → 개인화/추천이 더 필요한 영역
- 긴 쿼리(의도 명확) → 인공신경망 모델이 강점을 보이는 영역

**ANN과 Vector DB**

문서가 많아지면 전수 벡터 비교가 비현실적이므로 ANN(Approximate Nearest Neighbor)으로 근사 탐색한다.

- 주요 알고리즘: LSH, HNSW
- 대표 Vector DB: Elasticsearch, Faiss, Milvus, Pinecone

### 1.7 패러다임의 변화

**LLM의 등장**

2022년 ChatGPT 등장 이후 IR 분야에 큰 변화가 발생했다. LLM은 일반 상식 질의나 요약 작업에서 검색을 부분적으로 대체하기 시작했지만, 환각(hallucination)과 knowledge cutoff 문제가 여전히 존재한다.

**RAG (Retrieval-Augmented Generation)**

IR과 LLM을 결합한 대화형 검색 패러다임이다.

- IR로 관련 문서를 추출하고, LLM의 추론 능력으로 답변을 생성
- 대화 히스토리에서 독립 질의 생성 → 검색 → 적합성 판단 → 답변 생성의 파이프라인
- 검색 결과를 레퍼런스로 활용하여 사실 기반 정보를 대화형으로 제공

**변하지 않는 것: IR과 LLM의 역할 분담**

| 구분 | IR | LLM |
|------|-----|-----|
| 강점 | 휘발성 정보의 색인/검색, 조건 기반 빠른 추출 | 언어 패턴 이해, 의도 분석, 추론 기반 답변 생성 |
| 담당 영역 | 단순 지식, 실시간 갱신 정보 | 어휘, 공통 패턴, 추론 |

언어모델이 모든 지식을 내재화할 수 없으므로, 지속 갱신되는 정보는 학습이 아닌 검색(처리)의 대상으로 남는다.

## 2-1. 역색인과 형태소 분석기

### 1.1 문서 역색인

**색인(Indexing)** 이란 키워드 검색 시 문서를 빠르게 찾기 위해 문서 집합을 미리 가공하는 작업이다. 문서를 토큰 단위로 변환하여 빠른 추출이 가능한 구조로 저장한다.

**역색인(Inverted Index)** 은 일반적인 "문서 → 단어" 매핑을 뒤집어 "단어 → 문서 목록" 형태로 저장하는 구조다. 특정 키워드가 포함된 문서만 곧바로 추출할 수 있어 검색 속도가 빠르다.

**역색인 기반 색인 파이프라인:**

1. **텍스트 추출(Text Extraction)**: PDF, Word, HWP 등 다양한 형식에서 텍스트를 추출
2. **토큰화(Tokenization)**: 자연어 텍스트를 단어(토큰) 단위로 분해
3. **불용어 제거(Stop-word Removal)**: 검색에 의미 없는 관용어·불용어 제거 (예: the, is)
4. **정규화(Normalization)**: Lemmatization(표제어 추출)이나 Stemming(어간 추출)을 통해 단어를 기본 형태로 통일
5. **역색인 생성(Inverted Indexing)**: 최종 토큰을 key, 문서 목록을 value로 저장

### 1.2 형태소 분석기

**토큰화(Tokenization)** 는 문서나 질의를 정보의 기본 단위인 토큰으로 분리하는 과정이다. 문서 토큰은 역색인의 key로 저장되고, 질의 토큰은 역색인에서 탐색할 key로 사용된다.

성능 최적화를 위해 단순 토큰화 외에 불용어/품사 정보, 표제어 추출, 어간 추출, 복합명사 분리, 동의어 처리 등 추가 정보가 필요하다.

**토크나이저 종류:**

- **공백 기반(Whitespace)**: 공백으로 분리. 서유럽 언어에 효과적이나 복합어·구두점에 취약
- **규칙 기반(Rule-based)**: 정규 표현식이나 특정 규칙으로 분리
- **사전 기반(Dictionary-based)**: 미리 정의된 단어 사전을 활용. 한국어·중국어·일본어처럼 공백만으로 구분이 어려운 언어에 적합
- **서브워드(Subword)**: 단어를 더 작은 단위로 분리하는 학습 기반 방식. BPE, WordPiece, SentencePiece 등

### 1.3 한국어 형태소 분석의 특수성

한국어는 교착어로서 하나의 어근에 여러 형태소가 결합되며, 시제·존대·부정 등이 어미 변화로 표현된다. 복합명사가 많고 명사에 조사가 붙어 사용되며, 동일 철자가 다른 의미를 가지는 중의성 문제도 빈번하다.

따라서 한국어 토큰화 시에는 복합명사 분리, 조사 처리, 어간/어미 분리를 별도로 고려해야 한다.

**대표적인 한국어 형태소 분석기 (사전 기반):**

- **은전한닢(Seunjeon)**: Mecab-ko-dic 기반 오픈소스, JVM에서 구동. 적당한 품질과 속도
- **노리(Nori)**: Elastic이 2018년 출시한 Elasticsearch 공식 한국어 분석기. mecab-ko-dic 사전을 재가공하여 사용하며 빠른 분석 속도가 특징

## 2-2. Query-Document Relevance

### 2.1 Retrieval 모델

Query-Document Relevance를 계산하기 위한 세 가지 대표적인 Retrieval 모델을 다룬다.

#### 2.1.1 Boolean 모델

- 가장 단순한 검색 모델로, 질의를 AND/OR/NOT 등 논리 연산으로 표현하고 문서를 단어 집합으로 표현
- 조건을 만족하는 모든 문서를 반환하며, 결과에 대한 **순위를 매길 수 없음**
- 단어의 순서·중요도를 무시하고, 자연어를 직접 사용할 수 없어 단독 사용은 드묾

#### 2.1.2 Vector Space 모델

- 질의와 문서를 n차원 벡터 공간에 표현하고, 벡터 간 유사도를 측정하여 순위를 매기는 모델
- IR 시스템에서 가장 널리 사용됨
- **TF-IDF** 가중치 방식 사용:
  - **TF(t, d)**: 문서 d에서 단어 t가 등장한 빈도. 높을수록 해당 문서에서 중요한 단어일 가능성이 큼
  - **IDF(t, D)**: log(총 문서 수 / 단어 t를 포함하는 문서 수). 소수 문서에만 나타나는 단어일수록 정보 가치가 높음
  - TF-IDF(t, d, D) = TF(t, d) × IDF(t, D)
- **한계**: 철자 수준 매칭(동의어 미반영), 단어 순서 무시, 파라미터 최적화 비용 발생

#### 2.1.3 Probabilistic 모델

- 검색의 본질적 불확실성(질의 표현, 문서 표현, 매칭 방법)을 확률적으로 접근
- TF, IDF, 문서 길이 등 통계 자료를 활용하여 relevance 확률을 추정하고 log-odds로 랭킹
- **BIM(Binary Independence Model)**: 질의 단어의 존재 여부(0/1)만 사용, 단어 간 독립 가정
- **BM25**: BIM을 확장한 실용적 알고리즘으로, TF-IDF와 함께 검색 엔진에서 가장 널리 사용
  - **TF 개선**: 단어 빈도가 높아질수록 점수가 증가하지만 일정 한계치에 수렴(saturation)
  - **IDF 개선**: 너무 많은 문서에 등장하는 단어는 더욱 낮은(음수 가능) 점수 부여
  - **문서 길이 정규화**: 평균 문서 길이 대비 비율을 반영하여 길이에 따른 편향 완화
  - 수식: BM25(D, Q) = Σ IDF(qᵢ) · [TF(qᵢ, D) · (k₁+1)] / [TF(qᵢ, D) + k₁ · (1 - b + b · |D|/avgdl)]


### 2.2 Document Quality에 대한 고려

Retrieval 모델은 쿼리-문서 적합도만 고려하므로, 문서 품질까지 반영해야 최적의 결과를 제공할 수 있다.

#### 2.2.1 추가적인 Ranking 고려 사항

- 저품질·스팸·불법 복제 문서의 존재
- 시의성에 따른 문서 가치 변화 (예: 뉴스)

#### 2.2.2 주요 Ranking Factors

- **PageRank**: 하이퍼링크 구조를 directed graph로 모델링하여 페이지 중요도 측정. 많은 링크를 받거나 높은 점수의 페이지로부터 링크를 받을수록 높은 점수
- **Recency**: 최근 게시·업데이트된 문서에 높은 순위 부여. 예) Time Decay: 1/(1+age_in_days), Logarithmic Decay: 1/log(1+age_in_days)
- **Spam Score**: 키워드 스터핑, 저품질 인바운드/아웃바운드 링크, 메타 태그 남용, 콘텐츠 품질(문법 오류·표절 등) 등을 종합 평가


### 2.3 Relevance Feedback

#### 2.3.1 Relevance Feedback이 필요한 이유

- Retrieval 모델은 단어 통계 외의 요소를 무시하며, 문서 품질 계산도 처음부터 완벽할 수 없음
- 사용자 선호는 시간에 따라 변하므로, 실제 서비스 운영 중 피드백을 반영하여 랭킹을 지속 개선해야 함

#### 2.3.2 IRF(Implicit Relevance Feedback)의 적용

- 명시적 피드백을 강요하기 어려우므로, 클릭률·검색 기록·페이지 체류 시간 등 암묵적 사용자 행동 데이터를 활용
- 키워드-문서-IRF_score 형태의 triplet DB를 유지하고, 기존 ranking function에 IRF 가중치를 추가 반영
- **주의사항**: IRF 가중치가 과도하면 편향성 증가·다양성 감소 우려가 있으며, 사용자 프라이버시 및 개인정보 동의가 필요

## 2-3. 평가지표

### 3.1 Precision & Recall

#### 3.1.1 Precision & Recall 개념

IR 모델의 검색 결과는 실제 정답(Relevant)과 모델 추출(Retrieved)의 조합으로 네 가지로 분류: TP, FP, FN, TN

- **Precision** = TP / (TP + FP) — 추출된 문서 중 실제 관련 문서의 비율
- **Recall** = TP / (TP + FN) — 전체 관련 문서 중 추출된 문서의 비율
- **Accuracy** = (TP + TN) / 전체 문서 수

Precision과 Recall은 일반적으로 트레이드오프 관계에 있어, 둘 중 하나만으로는 모델 성능을 충분히 표현할 수 없다.

#### 3.1.2 추가적인 고려사항

- 대규모 문서 집합에서는 recall보다 상위 k개 결과가 중요 → **Precision@k** 활용
- **F1-score**: Precision과 Recall의 조화평균 = 2 × (P × R) / (P + R)
- **Average Precision (AP)**: Precision-Recall 곡선 아래 면적
- 관련성이 이분법(relevant/non-relevant)으로 나뉘지 않는 경우에 대한 고려 필요
- 신뢰도 확보를 위해 충분히 많은 질의 샘플에 대한 평균으로 평가

### 3.2 주요 IR Evaluation Metric

#### 3.2.1 MRR (Mean Reciprocal Rank)

- 각 질의에 대해 **첫 번째 관련 문서의 순위**에 역수를 취한 값(Reciprocal Rank)을 구하고, 전체 질의에 대해 평균을 계산
- 수식: MRR = (1/|Q|) × Σ (1/rankᵢ), 여기서 rankᵢ는 i번째 질의의 첫 번째 관련 문서 위치
- **적합한 경우**: 정답이 하나이거나 소수이고, 관련성이 이분법적으로 명확한 경우
- **한계**: 관련 문서가 여러 개인 상황에서는 첫 번째만 고려하므로 부적합

#### 3.2.2 MAP (Mean Average Precision)

- N개 질의에 대한 Average Precision(AP)의 평균
- AP 계산: 각 순위 k에서 관련 문서가 등장할 때의 Precision@k를 구하고, 이를 전체 관련 문서 수로 나눠 평균
  - AveP = Σ [P(k) × rel(k)] / 전체 관련 문서 수 (rel(k)는 k번째가 관련 문서이면 1, 아니면 0)
- **적합한 경우**: 관련 문서가 다수일 때, 상위 순위의 오류에 더 큰 패널티를 부여하고 싶을 때
- **한계**: MRR과 마찬가지로 관련성이 이분법적인 경우에만 적합하며, 관련도의 정도를 반영하지 못함

#### 3.2.3 NDCG (Normalized Discounted Cumulative Gain)

관련성을 이분법이 아닌 **다단계 점수**로 평가할 수 있는 지표로, 세 단계로 계산된다.

**① CG (Cumulative Gain)**
- 상위 K개 문서의 관련도 점수를 단순 합산
- CG_K = Σ relᵢ (i=1~K)

**② DCG (Discounted Cumulative Gain)**
- 순위가 낮을수록 할인(discount)을 적용하여, 관련 문서가 하위에 있을수록 점수가 감소
- DCG_K = Σ relᵢ / log₂(i+1) (i=1~K)

**③ NDCG (Normalized DCG)**
- DCG를 이상적인 순서(IDCG)로 정규화하여 0~1 사이 값으로 변환
- NDCG_K = DCG_K / IDCG_K
- IDCG는 관련도가 높은 순서대로 정렬했을 때의 DCG 값

**적합한 경우**: 관련도가 다양한 수치로 정량화 가능하고, 검색 결과의 다양성이 중요한 경우. 1에 가까울수록 이상적인 랭킹에 근접함을 의미한다.

**한계**: 관련도에 대한 다단계 레이블링이 필요하여 정답셋 구축 비용이 상대적으로 높다.

## 3-1. Elasticsearch 개념

### 1.1 Elasticsearch 기본 개념

#### 1.1.1 주요 특징

Elasticsearch는 실시간으로 대규모 데이터를 분석·검색할 수 있는 분산형 검색 및 분석 엔진이다.

- **분산형 아키텍처**: 데이터를 자동으로 여러 노드에 분산 저장하여 높은 가용성을 보장하고, 복제본을 통해 장애 시 데이터 손실을 방지
- **실시간 검색 및 분석**: 실시간 색인과 검색을 지원하며, 풀 텍스트 검색과 다양한 쿼리 언어를 통한 복잡한 검색이 가능
- **확장성 및 유연성**: 노드 추가로 손쉽게 수평 확장이 가능하며, 스키마리스(schema-less) 방식으로 다양한 유형의 데이터를 유연하게 처리
- **통합 솔루션**: Elastic Stack(Logstash, Kibana 등)과 결합하여 데이터 수집부터 시각화까지 full-stack을 지원

#### 1.1.2 RDBMS와의 용어 비교

| RDBMS | Elasticsearch |
|-------|---------------|
| Database | Search Engine |
| Table | Index |
| Row | Document |
| Column | Field |
| Schema | Mapping (type) |
| SQL | Query DSL |

Custom mapping을 생략하면 유입 데이터를 기반으로 동적으로 기본 스키마가 생성된다.

#### 1.1.3 데이터 구조

- **인덱스(Index)**: 독립적인 문서 구조를 가진 문서 집합의 저장 단위
- **타입(Type)**: 문서의 각 필드 구조(타입)
- **매핑(Mapping)**: 각 필드별 타입의 전체적인 매핑 구조 설정
- **문서(Document)**: 필드로 이루어진 색인의 단위 정보

#### 1.1.4 분산 컴포넌트 구조 (Sharding & Replication)

- **Shard**: 인덱스를 구성하는 물리적 단위. 인덱스가 샤드 단위로 분리되어 여러 노드에 분산 저장되며, 샤드 수 조정으로 검색 속도를 튜닝할 수 있다.
- **Replica**: 처리량 증가와 노드 장애 시 데이터 신뢰성 확보를 위해 샤드를 복제한 것. 서로 다른 노드에 골고루 분배되어 저장된다.


### 1.2 문서 색인

#### 1.2.1 색인 설정과 매핑

- **Dynamic Mapping**: 별도 설정 없이 유입 데이터를 기반으로 자동 매핑 생성
- **Explicit Mapping**: 필드별로 원하는 타입(integer, keyword, text 등)을 명시적으로 설정하여 색인

#### 1.2.2 형태소 분석기 활용

한국어의 경우 기본 토크나이저로는 원하는 검색 성능을 얻기 어려우므로 형태소 분석기를 사용해야 한다. Elasticsearch에서는 한국어 형태소 분석기 **Nori**를 제공하며, nori_tokenizer와 nori_part_of_speech 필터를 조합하여 어미, 조사, 구분자 등 불필요한 품사를 제거할 수 있다.


### 1.3 문서 검색

#### 1.3.1 Query DSL

Elasticsearch는 JSON 기반의 Query DSL(Domain Specific Language)을 사용하여 쿼리를 정의한다.

- **구문 타입** (트리 형태로 구성 가능):
  - Leaf query clauses — 특정 필드에 대해 원하는 조건으로 검색 수행
  - Compound query clauses — Leaf 또는 다른 Compound 절을 감싸는 복합 구문
- **사용 형태**:
  - Query context — 문서 추출 시 relevance score를 함께 계산
  - Filter context — 조건에 맞는 문서만 추출하고 score 계산은 하지 않음
- **Expensive query**: fuzzy, regex, prefix, wildcard, range, script_score 등 추가 지원

#### 1.3.2 검색 결과 확인

Query DSL로 구성한 쿼리를 REST API로 호출하여 결과를 얻는다.

- **요청 시 옵션**: 대상 인덱스, 검색 결과 최대 개수, timeout 등
- **응답 정보**: Top-k 적합 문서 리스트, 문서별 relevance score, 전체 매칭 문서 수, 샤드·timeout 등 메타 정보

## 4-1. Vector Similarity

### 1.1 임베딩 생성

#### 1.1.1 임베딩 방식의 진화 과정

**임베딩이란?**
- 단어나 문장을 고차원의 실수 벡터로 변환하는 기법
- 벡터 간 산술 연산이 가능하며, 이를 통해 단어·문장 간 유사도를 정량적으로 측정할 수 있음

**Word2Vec**
- 학습 기반 임베딩의 초기 대표 모델
- 핵심 아이디어: 함께 등장하는(co-occurring) 단어들이 벡터 공간에서 가까이 위치하도록 학습
- 두 가지 구조: CBOW(주변 단어 → 중심 단어 예측), Skip-gram(중심 단어 → 주변 단어 예측)

**Word2Vec 확장 모델들**

| 모델 | 핵심 아이디어 |
|------|------------|
| Doc2Vec | 문장·문단·문서 단위까지 벡터화. 문서 ID를 추가 context로 활용 |
| Top2Vec | 문서·단어 임베딩을 함께 사용하여 토픽 임베딩 생성. 차원 축소 후 클러스터 중심을 토픽 벡터로 활용 |
| BioVectors | 단백질·DNA 등 생물학적 시퀀스에 n-gram 방식 적용 (ProtVec, GeneVec 등) |

**컨텍스트 기반 임베딩의 등장**
- Word2Vec의 한계: 단어마다 하나의 고정된 벡터만 존재 → 다의어(polysemy) 처리 불가
- 해결: 주변 문맥에 따라 동적으로 임베딩을 생성하는 방식으로 발전

| 모델 | 특징 |
|------|------|
| ELMo | 문맥 고려 임베딩의 시초. LSTM 기반 양방향 언어 모델 |
| ULM-FiT | pre-training → fine-tuning 패러다임을 확립. 다양한 downstream task 적용 가능 |
| OpenAI Transformer | Transformer decoder 구조. 다음 토큰 예측(autoregressive) 방식 |
| BERT | Transformer encoder 구조. 양방향 문맥을 동시에 반영하여 높은 성능 달성 |

#### 1.1.2 Pre-trained 모델 활용

- BERT는 임베딩 생성뿐 아니라 다양한 NLP task에 범용적으로 활용 가능
  - 문장 쌍 분류 (MNLI, QQP, STS-B 등)
  - 단일 문장 분류 (SST-2, CoLA)
  - 질의응답 (SQuAD)
  - 개체명 인식 (CoNLL NER) 등


### 1.2 Vector Similarity

#### 1.2.1 IR을 위한 임베딩 활용

**Pre-trained BERT를 직접 사용하는 방식**
- CLS 토큰 임베딩 또는 전체 토큰 임베딩의 평균값을 문서·질의의 벡터 표현으로 사용
- 질의와 문서를 각각 독립적으로 인코딩한 뒤 벡터 유사도(코사인 유사도 등)를 계산
- **한계**: 질의-문서 간 유사도를 직접 학습한 것이 아니므로 검색 성능이 제한적

**Fine-tuning을 통한 개선 (Cross-Encoder)**
- 질의와 문서를 하나의 입력으로 결합: `[CLS] query [SEP] document [SEP]`
- BERT 내부에서 질의-문서 간 상호작용(cross-attention)이 발생
- CLS 토큰 출력에 MLP를 붙여 관련도 점수를 산출
- 질의-문서 pair 데이터로 fine-tuning하면 유사도 성능이 크게 향상됨

#### 1.2.2 기본 모델의 한계

BERT cross-encoder 방식은 정확도는 높지만 실용적 한계가 존재:

1. **학습 데이터 구축 비용**: 고품질 질의-문서 pair 데이터를 대량으로 만드는 데 높은 비용 소요
2. **추론 속도 문제**: 모든 질의-문서 조합을 모델에 통과시켜야 하므로 inference 비용이 매우 큼. 문서 수가 수천 건만 넘어도 실시간 서비스가 어려움
3. **실서비스 적용의 어려움**: cross-encoder는 주로 reranking 단계에서 소규모로 활용되며, 대규모 검색에는 DPR·SentenceBERT 등 **bi-encoder** 방식이 사용됨

> **Cross-Encoder vs Bi-Encoder**
> - Cross-Encoder: 질의+문서를 하나의 모델에 함께 입력 → 높은 정확도, 느린 속도
> - Bi-Encoder: 질의와 문서를 별도 인코더로 독립 임베딩 → 사전 인덱싱 가능, 빠른 검색

## 4-2. Query-Document 매칭 알고리즘

### 2.1 Cross-Encoder 모델 구조

#### 2.1.1 기본 구조
##### 2.1.1.1 입력 방식
- BERT 모델에 질의와 문서를 하나의 시퀀스로 결합하여 입력
- 입력 형식: `[CLS] query tokens [SEP] document tokens [SEP]`
- 모델 내부에서 질의 토큰과 문서 토큰 간 cross-attention이 발생

##### 2.1.1.2 점수 산출
- [CLS] 토큰의 출력 임베딩 또는 전체 토큰 평균값을 활용
- MLP를 통과시켜 최종 relevance score 산출

#### 2.1.2 학습 방법
##### 2.1.2.1 학습 데이터 구성
- triplet 형태: `⟨query, positive_doc, {negative_docs}⟩`
- positive pair의 점수는 높이고, negative pair의 점수는 낮추는 contrastive learning 방식

### 2.2 Cross-Encoder 모델 특징

#### 2.2.1 성능과 속도의 Trade-off
##### 2.2.1.1 장점 — 높은 정확도
- 질의-문서 토큰 간 직접적인 attention이 이루어져 매칭 성능이 매우 높음
- 충분한 학습 데이터가 확보되면 성능을 극대화할 수 있음

##### 2.2.1.2 단점 — 느린 추론 속도
- 질의마다 전체 문서에 대해 각각 모델 inference를 수행해야 함
- 응답 시간이 문서 수에 비례하여 선형 증가
- 대규모 서비스에서 직접 검색 단계에 적용하기 어려움

### 2.3 Bi-Encoder 모델 구조

#### 2.3.1 기본 구조
##### 2.3.1.1 독립 인코딩
- 질의와 문서를 별도의 인코더에 각각 독립적으로 입력하여 임베딩 생성
- 학습 시 질의 토큰과 문서 토큰 사이에 직접적인 attention 연결 없음
- 최종 유사도 점수만으로 두 인코더가 동시에 업데이트됨

##### 2.3.1.2 파라미터 공유 옵션
- 논리적으로는 두 개의 독립 모델이지만, 물리적으로 파라미터를 공유하는 단일 모델로 구현할 수도 있음

#### 2.3.2 유사도 계산
##### 2.3.2.1 계산 방식
- 각 인코더에서 CLS 토큰의 출력 임베딩을 추출
- 질의 임베딩과 문서 임베딩 간 내적(dot product) 또는 코사인 유사도로 관련도 점수 산출

### 2.4 Bi-Encoder 모델 특징

#### 2.4.1 성능과 속도의 Trade-off
##### 2.4.1.1 장점 — 빠른 검색 속도
- 문서 임베딩을 오프라인에서 사전 계산(pre-compute)하여 인덱싱 가능
- 검색 시에는 질의 임베딩만 실시간 생성 후 벡터 유사도 계산만 수행

##### 2.4.1.2 단점 — 상대적으로 낮은 정확도
- 질의-문서 간 토큰 레벨 상호작용이 없어 cross-encoder 대비 성능이 떨어짐
- 다만 잘 학습된 경우 sparse retrieval(BM25 등)보다는 일반적으로 우수

##### 2.4.1.3 실용성
- 연산 비용과 속도 면에서 대규모 검색 서비스에 현실적으로 적용 가능

### 2.5 Dense Passage Retrieval (DPR)

#### 2.5.1 개요
##### 2.5.1.1 모델 구조
- Bi-encoder의 대표적인 모델
- 질의 인코더(EncQ)와 문서 인코더(EncD)를 각각 독립 파라미터를 가진 BERT로 구성

#### 2.5.2 학습 과정
##### 2.5.2.1 유사도 계산
- 각 인코더 출력의 CLS 임베딩 간 내적: `Sim(q, doc) = EncQ(q)ᵀ · EncD(doc)`

##### 2.5.2.2 손실 함수
- negative log-likelihood 기반
- positive pair의 유사도는 극대화, negative pair의 유사도는 최소화하는 방향으로 두 인코더를 동시에 학습

### Cross-Encoder vs Bi-Encoder 비교 요약

| 구분 | Cross-Encoder | Bi-Encoder |
|------|--------------|------------|
| 입력 방식 | 질의+문서를 결합하여 단일 모델에 입력 | 질의와 문서를 별도 인코더에 각각 입력 |
| 토큰 간 상호작용 | 있음 (cross-attention) | 없음 (독립 인코딩) |
| 문서 사전 인덱싱 | 불가능 | 가능 |
| 검색 정확도 | 높음 | 상대적으로 낮음 |
| 추론 속도 | 느림 (문서 수에 비례) | 빠름 (사전 인덱싱 활용) |
| 실서비스 적용 | reranking 단계에 제한적 활용 | 1차 retrieval 단계에 적합 |
